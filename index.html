I can hold the technical conversation, not just translate it

An enablement specialist who cannot evaluate the technology ends up relaying vendor claims. I work from how these systems actually behave — what retrieval does and does not guarantee, where a model fails, and how you would prove it on your own matters before anyone stakes work on it.

01 Retrieval, grounding, and why “it cited something” is not enough

Most legal AI is retrieval-augmented: the model answers from documents fetched at query time, so the quality of the answer is bounded by the quality of the retrieval. That means the failure mode worth training people on is not only the invented citation — it is the confident answer built on the wrong or incomplete set of retrieved documents. I teach practitioners to check what the tool actually looked at, not just whether the output looks authoritative.

02 Evaluating a tool against real work, not a demo

A vendor demo is a curated best case. A useful evaluation runs the tool on representative matters you already know the answers to, and scores it: was it accurate, was it checkable, did it save real time, did it hold up on the messy inputs rather than the clean ones? A gold-set of known-answer matters turns “the attorneys liked it” into evidence — and gives you a baseline to re-test against when the model changes underneath you.

03 Prompt and workflow design as an engineering practice

Prompting is not incantation. It is specification: give the model the role, the source material, the constraints, the output shape, and a place to say “I don’t know.” The durable artifact is not a clever prompt — it is a documented workflow with a human-review checkpoint at the step where being wrong is expensive. I build those, run them on real work, and hand them over so someone else can run them too.
